# Command Reference

This comprehensive reference covers all PruneJuice CLI commands, their arguments, options, and usage patterns.

## Quick Reference

### Core Commands
- `prj init [PATH]` - Initialize a new PruneJuice project
- `prj run COMMAND [ARGS...]` - Execute a command
- `prj list [TYPE]` - List events or commands
- `prj status` - Show project status
- `prj history` - Show command execution history
- `prj show EVENT_ID` - Show detailed event information
- `prj cleanup` - Clean up old artifacts

### Subcommand Groups
- `prj worktree` - Git worktree management
- `prj session` - Tmux session management

---

## Core CLI Commands

### `prj init`

Initialize a PruneJuice project in the current or specified directory.

**Usage:**
```bash
prj init [PATH]
```

**Arguments:**
- `PATH` (optional) - Project path to initialize (defaults to current directory)

**What it does:**
1. Creates `.prj/` directory structure
2. Initializes SQLite database
3. Copies template commands and steps
4. Sets up artifact storage

**Directory Structure Created:**
```
.prj/
â”œâ”€â”€ commands/          # Command definitions (YAML)

â”œâ”€â”€ artifacts/        # Command execution artifacts
â””â”€â”€ prunejuice.db     # SQLite database
```

**Example:**
```bash
# Initialize current directory
prj init

# Initialize specific path
prj init /path/to/project
```

**Return Codes:**
- `0` - Success
- `1` - Error during initialization

### `prj run`

Execute a command with optional arguments.

**Usage:**
```bash
prj run COMMAND [ARG=VALUE...] [OPTIONS]
```

**Arguments:**
- `COMMAND` - Name of the command to execute
- `ARG=VALUE` - Command arguments in key=value format

**Options:**
- `--dry-run` - Show what would be executed without running
- `--verbose, -v` - Enable verbose output
- `--help` - Show help message

**Examples:**
```bash
# Simple command
prj run echo-hello

# Command with arguments
prj run echo-arg message="Hello World"

# Multiple arguments
prj run analyze-issue issue_number=123 branch_name=fix-bug

# Dry run to see what would execute
prj run code-review --dry-run

# Verbose output
prj run feature-branch feature_name=new-auth --verbose
```

**Return Codes:**
- `0` - Command completed successfully
- `1` - Command failed or error occurred

### `prj list`

List recent events or available commands.

**Usage:**
```bash
prj list [TYPE] [OPTIONS]
```

**Arguments:**
- `TYPE` - What to list: `events` (default) or `commands`

**Options:**
- `--limit, -n NUMBER` - Number of events to show (events only, default: 10)
- `--status STATUS` - Filter by status: `completed`, `failed`, `running`, `cancelled`
- `--verbose, -v` - Show detailed information

**Examples:**
```bash
# List recent events (default)
prj list
prj list events

# List last 20 events
prj list events --limit 20

# Filter by status
prj list events --status failed

# List available commands
prj list commands

# Verbose command listing
prj list commands --verbose
```

### `prj history`

Show command execution history with advanced filtering options.

**Usage:**
```bash
prj history [OPTIONS]
```

**Options:**
- `--limit, -n NUMBER` - Number of events to show (default: 20)
- `--status STATUS` - Filter by status: `completed`, `failed`, `running`, `cancelled`
- `--command COMMAND` - Filter by command name
- `--worktree WORKTREE` - Filter by worktree name
- `--project` - Filter to current project only

**Examples:**
```bash
# Show last 50 events
prj history --limit 50

# Show only failed commands
prj history --status failed

# Show history for specific command
prj history --command code-review

# Show history for current project only
prj history --project

# Combined filters
prj history --status completed --command echo-hello --limit 10
```

### `prj cleanup`

Clean up old artifacts and sessions.

**Usage:**
```bash
prj cleanup [OPTIONS]
```

**Options:**
- `--help` - Show help message
- `--yes, -y` - Skip confirmation prompt
- `--days DAYS` - Clean up artifacts older than N days (default: 30)

**Examples:**
```bash
# Clean up with confirmation
prj cleanup

# Clean up artifacts older than 7 days
prj cleanup --days 7

# Skip confirmation
prj cleanup --yes
```

### `prj worktree create`

Create a new worktree with the specified branch.

**Usage:**
```bash
prj worktree create BRANCH_NAME [OPTIONS]
```

**Arguments:**
- `BRANCH_NAME` - Name for the new branch and worktree

**Options:**
- `--base, -b BASE_BRANCH` - Base branch to create from (optional)

**Examples:**
```bash
# Create feature branch
prj worktree create feature-auth

# Create from specific base branch
prj worktree create bugfix-login --base develop
```

---

### `prj worktree remove`

Remove an existing worktree.

**Usage:**
```bash
prj worktree remove WORKTREE_PATH [OPTIONS]
```

**Arguments:**
- `WORKTREE_PATH` - Path to the worktree to remove

**Options:**
- `--force, -f` - Force removal without confirmation

**Examples:**
```bash
# Remove with confirmation
prj worktree remove /path/to/worktree

# Force remove
prj worktree remove /path/to/worktree --force
```

## Session Commands

The `prj session` subcommand group manages tmux sessions through the pots integration.

### `prj session list`

List all tmux sessions managed by pots.

**Usage:**
```bash
prj session list [OPTIONS]
```

**Options:**
- `--verbose, -v` - Show detailed session information including status

**Examples:**
```bash
# Basic listing
prj session list

# Detailed listing
prj session list --verbose
```

---

### `prj session create`

Create a new tmux session for a worktree.

**Usage:**
```bash
prj session create TASK_NAME [OPTIONS]
```

**Arguments:**
- `TASK_NAME` - Name for the new session/task

**Options:**
- `--dir, -d WORKING_DIR` - Worktree directory (default: current directory)

**Note:** The working directory must be a valid git worktree for pots to create a session.

**Examples:**
```bash
# Create in current directory
prj session create my-task

# Create in specific worktree
prj session create auth-work --dir /path/to/worktree
```

---

### `prj session attach`

Attach to an existing tmux session.

**Usage:**
```bash
prj session attach SESSION_NAME
```

**Arguments:**
- `SESSION_NAME` - Name of the session to attach to

**Examples:**
```bash
prj session attach my-task
```

---

### `prj session kill`

Kill an existing tmux session.

**Usage:**
```bash
prj session kill SESSION_NAME [OPTIONS]
```

**Arguments:**
- `SESSION_NAME` - Name of the session to kill

**Options:**
- `--force, -f` - Force kill without confirmation

**Examples:**
```bash
# Kill with confirmation
prj session kill my-task

# Force kill
prj session kill my-task --force
```

---

### `prj session cleanup`

Clean up inactive or orphaned sessions.

**Usage:**
```bash
prj session cleanup [OPTIONS]
```

**Options:**
- `--dry-run` - Show what would be cleaned up without actually removing
- `--force, -f` - Force cleanup without confirmation

**Examples:**
```bash
# Show what would be cleaned
prj session cleanup --dry-run

# Clean up with confirmation
prj session cleanup

# Force cleanup
prj session cleanup --force
```

## Template Commands

PruneJuice includes several template commands that are copied to new projects during initialization. These demonstrate different command patterns and can be customized for your needs.

### Template Command Summary

| Command | Category | Description | Key Arguments | What it Creates |
|---------|----------|-------------|---------------|-----------------|
| `echo-hello` | example | Simple echo command | None | Basic output test |
| `echo-arg` | example | Echo with arguments | `message` (required) | Argument handling demo |
| `worktree-list` | example | List worktrees via script | None | Script execution example |
| `echo-hello-in-session` | example | Session workflow demo | None | Tmux session lifecycle |
| `analyze-issue` | analysis | Analyze GitHub issue | `issue_number` (required), `branch_name` (optional) | Analysis workflow |
| `code-review` | quality | Comprehensive code review | `target_branch` (optional), `focus_areas` (optional) | Review workflow |
| `feature-branch` | development | Feature development setup | `feature_name` (required), `issue_number` (optional) | Development environment |

### Template Usage Examples

```bash
# Simple examples
prj run echo-hello
prj run echo-arg message="Hello PruneJuice!"
prj run worktree-list

# Session workflow
prj run echo-hello-in-session

# Development workflows
prj run analyze-issue issue_number=123
prj run code-review target_branch=develop focus_areas=security
prj run feature-branch feature_name=user-auth issue_number=456
```

### Worktree Operations
- `prj worktree list` - View active worktrees
- `prj worktree create` - Create new worktrees
- `prj worktree remove` - Clean up worktrees

- `prj session list` - View active sessions
- `prj session create` - Start new sessions
- `prj session attach` - Connect to sessions
- `prj session kill` - End sessions
- `prj session cleanup` - Maintain sessions

### Development Utilities
- Template commands for common workflows
- Integration with external tools (plum, pots)
- Artifact management and storage

## Advanced Usage

### Environment Variables

PruneJuice supports configuration through environment variables with the `PRUNEJUICE_` prefix:

```bash
# Database and storage
export PRUNEJUICE_DB_PATH="/custom/path/to/database.db"
export PRUNEJUICE_ARTIFACTS_DIR="/custom/artifacts"

# Integration paths
export PRUNEJUICE_PLUM_PATH="/usr/local/bin/plum"
export PRUNEJUICE_POTS_PATH="/usr/local/bin/pots"

# Execution settings
export PRUNEJUICE_DEFAULT_TIMEOUT=3600
export PRUNEJUICE_MAX_PARALLEL_STEPS=1

# Development settings
export PRUNEJUICE_GITHUB_USERNAME="yourusername"
export PRUNEJUICE_EDITOR="vim"
export PRUNEJUICE_BASE_DIR="/workspace"
```

### Configuration Files

PruneJuice looks for configuration in `.env` files:

```bash
# .env file in project root
PRUNEJUICE_DEFAULT_TIMEOUT=2400
PRUNEJUICE_GITHUB_USERNAME=myuser
PRUNEJUICE_EDITOR=code
```

### Return Codes

All commands use consistent return codes:

- `0` - Success
- `1` - General error or command failure

### Integration Patterns

**With Git Hooks:**
```bash
# In .git/hooks/pre-commit
prj run code-review --dry-run
```

**With CI/CD:**
```bash
# In CI pipeline
prj init
prj run analyze-issue issue_number=$ISSUE_NUMBER
```

**With Development Workflows:**
```bash
# Create feature branch workflow
prj worktree create feature-$FEATURE_NAME
cd ../project-feature-$FEATURE_NAME
prj session create $FEATURE_NAME
prj run feature-branch feature_name=$FEATURE_NAME
```

### Pots Integration  
- **Purpose**: Tmux session management
- **Installation**: Required for `prj session` commands
- **Requirements**: tmux must be available
- **Availability**: Check with `prj status`
